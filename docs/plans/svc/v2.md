# SVC Proof-of-Concept with Wegmans Data - Plan v2

## Goal

Validate the SVC approach using the Wegmans dataset (24,833 products) before investing in additional scrapers. Train a Direct Linear SVC that classifies products into the app's categories, running entirely in the browser for offline use.

### The Bigger Picture

SuperSuper's end goal is to reduce food waste by helping users avoid duplicate purchases. When a user is about to buy something, the app should be able to:
- Categorize the product accurately (existing feature, dramatically improved by direct SVC)
- **Suggest similar products already in the user's pantry** (future feature, enabled by SVC embeddings)

For example, if the user is about to buy "Wegmans Italian Parmigiano Reggiano Cheese", the app should notice they already have "Kraft Parmesan Cheese" in their pantry and flag it.

### Why v2 (Proof-of-Concept First)

v1 planned to scrape 4 supermarkets before training. This plan front-loads SVC validation using only Wegmans data. **Result: the direct SVC achieves 82.4% test accuracy vs 50.3% baseline -- a +32pp improvement.** This confirms the approach works and justifies proceeding with more data.

## Results Summary

| Approach | Accuracy | Delta vs Baseline |
|----------|----------|-------------------|
| Baseline (cosine similarity) | 50.3% | -- |
| Feature weights (L2 norm) | 50.3% | +0.0pp |
| Projected embeddings | 47.2% | -3.1pp |
| **Direct SVC (winner)** | **82.4% test / 85.5% full** | **+32.1pp** |

### Key Finding: Feature Weights Don't Work

The original hypothesis was that extracting per-dimension importance weights from the SVC and applying them to cosine similarity would improve classification. **This was tested and disproven:**

- L2 norm of SVC weight columns as feature weights: 0% improvement
- Projecting embeddings through the weight matrix: actually worse (-3.1pp)

The reason: L2 norm discards directional information. The SVC's power comes from its full weight matrix (13 classes x 384 features) and bias vector, not from a scalar importance per dimension.

### What Works: Direct SVC Classification

Training a Linear SVC directly on the app's 13 categories (mapped from Wegmans' 105 native categories) produces a dramatic improvement. The model is a simple linear classifier: `category = argmax(embedding @ W.T + bias)`.

**Per-category improvements (selected):**
- Pets: +88.6pp (10.0% -> 98.6%)
- Home & outdoor: +66.6pp (21.3% -> 87.9%)
- Deli & prepared food: +51.4pp (5.0% -> 56.4%)
- Baby & kids: +43.8pp (45.4% -> 89.2%)
- Health & beauty: +40.1pp (51.9% -> 91.9%)

### Category Mapping IS Required for Training

Contrary to the original plan, the direct SVC approach requires mapping each store's native categories to the app's categories. This is a one-time effort per store (Wegmans' 105 categories have been mapped). The mapping exists in `evaluate_baseline.py` as `EVAL_MAPPING`. When additional stores are added, their categories must also be mapped.

## How It Works: Direct SVC Classification

### The Problem with the Previous Approach

The previous classifier computed cosine similarity between a product embedding and each subcategory embedding. All 384 dimensions were weighted equally, and the general-purpose model (`all-MiniLM-L6-v2`) doesn't know which dimensions matter for grocery products. This resulted in only 50.3% accuracy.

### How the Direct SVC Fixes This

1. **Train on mapped grocery data**: Map Wegmans' 105 native leaf categories to the app's 13 categories. Train a Linear SVC on the mapped labels using the 384-dim embeddings.

2. **Export the full decision function**: The trained SVC gives a weight matrix (13 x 384) and bias vector (13). These are exported as JSON for browser use.

3. **Classify at runtime**: For a new product, compute its embedding (existing pipeline), then: `scores = embedding @ weights.T + bias; category = classes[argmax(scores)]`. This is a single matrix-vector multiplication -- extremely fast.

### Runtime Flow (Browser)

```
New product name
  -> embedding via all-MiniLM-L6-v2 (existing, ~23MB, cached)
  -> scores = embedding * weights_matrix + bias  (100KB model, cached)
  -> argmax(scores) -> category
  -> fallback: cosine similarity if SVC model unavailable
```

## Offline Constraint

The SVC model is fully usable offline:

1. **Embedding model** (`all-MiniLM-L6-v2`, ~23MB): Already client-side via `@xenova/transformers`, cached by browser. No change needed.
2. **SVC model** (`svc_model.json`, 100.5 KB): Served from `public/models/`, cached by service worker on install via `CACHE_URLS`.
3. **Classification at runtime**: Product name -> embedding (existing) -> matrix multiply + bias -> argmax -> category. No server calls needed.

## Phases

### Phase 1: Embedding Generation [DONE]

Generated 384-dim embeddings for all 24,833 Wegmans products using `sentence-transformers` with `all-MiniLM-L6-v2` (same model as browser). Also generated embeddings for the app's ~157 subcategory names.

**Artifacts:**
- `data/embeddings/product_embeddings.npy` (24,833 x 384 matrix)
- `data/embeddings/product_labels.json` (category labels)
- `data/embeddings/subcategory_embeddings.npy` (157 x 384)
- Generation time: ~164 seconds

### Phase 2: Baseline Evaluation [DONE]

Reimplemented the browser's `categoryClassification.js` logic in Python and evaluated against Wegmans data.

**Result: 50.3% accuracy** -- the baseline against which all approaches are measured.

The evaluation requires a category mapping (Wegmans leaf category -> app category) defined as `EVAL_MAPPING` in `evaluate_baseline.py`. This same mapping is reused for training the direct SVC.

### Phase 3: SVC Training [DONE]

Trained a Linear SVC on native Wegmans categories (105 classes) to explore the feature-weights approach.

**Result: 83.2% test accuracy on native categories.** However, extracting feature weights from this model did NOT improve classification against the app's categories (see Phase 4).

### Phase 4: Approach Comparison [DONE]

Tested three approaches against the baseline:

1. **Feature weights** (L2 norm of SVC weight columns): 50.3% -- no improvement
2. **Projected embeddings** (through 105-class weight matrix): 47.2% -- worse
3. **Direct SVC** (trained on mapped app categories): **82.4% test / 85.5% full**

The direct SVC was trained with hyperparameter search (C values: 0.01, 0.1, 1.0, 10.0 via 5-fold CV). Best: C=10.0.

**Decision: Direct SVC is the clear winner.** The feature-weights hypothesis was disproven.

### Phase 5: Export and Browser Integration [DONE]

**Export:**
- `export_model.py` exports the direct SVC's weight matrix (13 x 384), bias (13), and class labels to `public/models/svc_model.json` (100.5 KB)

**Code changes:**
- `src/services/categoryClassification.js`:
  - On init: loads embedding model + fetches `svc_model.json`
  - `_loadSvcModel()`: Parses weight matrix, bias, and class labels
  - `_classifySvc()`: `scores = embedding @ W.T + bias; argmax -> category`
  - `_classifyCosine()`: Original cosine similarity approach, used as fallback
  - `classifyProduct()`: Routes to SVC or cosine based on model availability
  - `getEmbedding()`: Public method exposing raw 384-dim embeddings for future similarity use
  - `getStatus()`: Reports SVC mode and class count
- `public/sw.js`: Added `/models/svc_model.json` to `CACHE_URLS`

**Fallback behavior:** If the SVC model fails to load (network error, corrupted JSON), the service gracefully falls back to the original cosine similarity approach. No degradation from the user's perspective -- they just get the old accuracy instead of the improved SVC accuracy.

## Notes

- **Category mapping IS required for training**: The direct SVC must be trained on mapped labels (store categories -> app categories). This is a one-time effort per store. The mapping for Wegmans (105 categories) exists in `evaluate_baseline.py` as `EVAL_MAPPING`.
- **Feature weights approach was disproven**: The original hypothesis that L2-norm feature weights would improve cosine similarity was tested and found to provide 0% improvement. The full decision function (matrix multiply + bias) is needed.
- **Same embedding model**: Both baseline and SVC approaches use `all-MiniLM-L6-v2`. The SVC adds a learned linear layer on top of the existing embeddings.
- **No new browser dependencies**: The SVC prediction is plain JavaScript (nested loops for matrix multiply). No ML libraries needed.
- **Future: product similarity**: The `getEmbedding()` method exposes raw embeddings for product-to-product comparison. For fine-grained similarity, the full 384-dim embeddings are more useful than the 13-dim SVC projections. A separate similarity model may be needed for the duplicate detection feature.
- **Additive with more data**: When HEB, Walmart, Kroger data is added (v1 Phase 2+), the direct SVC can be retrained with more products. The browser-side code doesn't change -- just deploy a new `svc_model.json`.
- **Food waste reduction**: Better categorization is the foundation. The 32pp accuracy improvement means products are placed in the correct category far more often, making pantry browsing and duplicate detection more reliable.
