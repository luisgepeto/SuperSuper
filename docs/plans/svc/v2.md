# SVC Proof-of-Concept with Wegmans Data - Plan v2

## Goal

Validate the SVC approach using the Wegmans dataset (24,833 products) before investing in additional scrapers. Train a Linear SVC on real supermarket data to **learn which embedding dimensions matter for grocery products**, then use those learned weights to improve both:

1. **Product categorization** -- scoring products against the app's existing category structure
2. **Product similarity** -- identifying similar products to reduce food waste by preventing duplicate purchases

### The Bigger Picture

SuperSuper's end goal is to reduce food waste by helping users avoid duplicate purchases. When a user is about to buy something, the app should be able to:
- Categorize the product accurately (existing feature, improved by SVC weights)
- **Suggest similar products already in the user's pantry** (future feature, enabled by the same SVC weights)

For example, if the user is about to buy "Wegmans Italian Parmigiano Reggiano Cheese", the app should notice they already have "Kraft Parmesan Cheese" in their pantry and flag it. This requires accurate product-to-product similarity, not just product-to-category matching.

The general-purpose embedding model (`all-MiniLM-L6-v2`) treats all semantic dimensions equally -- it might weight the word "Italian" as heavily as "Cheese" when comparing groceries. The SVC learns from real grocery data which dimensions actually matter for distinguishing food products, improving both categorization and similarity.

### Why v2 (Proof-of-Concept First)

v1 planned to scrape 4 supermarkets before training. This plan front-loads the SVC validation using only Wegmans data to answer the key question: **does an SVC with learned weights outperform the current zero-shot cosine similarity?** If yes, we proceed with more data (v1 Phase 2+). If not, we reassess before spending time on additional scrapers.

## How It Works: SVC-Weighted Scoring

### The Problem with Current Approach

The current classifier computes cosine similarity between a product embedding and each subcategory embedding. All 384 dimensions of the embedding are weighted equally. But in the grocery domain, some dimensions capture information that is far more relevant to food categorization than others. The model (`all-MiniLM-L6-v2`) is a general-purpose sentence encoder -- it doesn't know that grocery-related features should be prioritized.

This same problem affects product-to-product similarity. When comparing "Parmigiano Reggiano" to "Kraft Parmesan", the general-purpose model might focus on brand names or descriptive words rather than the food characteristics that make them functionally similar products.

### How the SVC Fixes This

1. **Train on real grocery data**: We train a Linear SVC using Wegmans' native category structure (105 leaf categories). The SVC learns from real product-to-category associations which embedding dimensions best distinguish grocery categories.

2. **Extract learned feature weights**: The trained SVC's weight matrix captures per-dimension importance for the grocery domain. We derive a **feature importance vector** (384 weights) that tells us which dimensions of the embedding space matter most for grocery products.

3. **Apply weights at scoring time**: When classifying or comparing products, we multiply embeddings by these learned weights before computing cosine similarity. This "focuses" the similarity computation on the dimensions that actually matter for grocery products.

### Two Use Cases, One Set of Weights

The same feature weights improve both:

| Use Case | Current Approach | With SVC Weights |
|----------|-----------------|------------------|
| **Categorization** | `similarity(product, subcategory)` -- all dimensions equal | `similarity(product * W, subcategory * W)` -- grocery dims emphasized |
| **Product Similarity** | `similarity(productA, productB)` -- all dimensions equal | `similarity(productA * W, productB * W)` -- food-relevant dims emphasized |

For duplicate detection, the weighted similarity between "Parmigiano Reggiano" and "Kraft Parmesan" should be higher than the unweighted similarity, because the weights emphasize cheese/dairy dimensions over brand/marketing dimensions.

### The Key Insight

We do NOT need to map Wegmans categories to our app categories. The SVC trains on Wegmans' own category structure to learn general grocery-domain feature weights. These weights improve similarity scoring against ANY set of category labels -- including our app's existing 12 categories. When more supermarket data is added later (HEB, Walmart, Kroger), it further refines these weights by exposing the model to more diverse product-category relationships.

### Runtime Flow (Browser)

```
New product name
  -> embedding via all-MiniLM-L6-v2 (existing, ~23MB, cached)
  -> multiply embedding by learned SVC weights (384 floats, <2KB)
  -> cosine similarity against weighted subcategory embeddings
  -> best match -> category
```

## Offline Constraint

The SVC model must be fully usable offline. This means:

1. **Embedding model** (`all-MiniLM-L6-v2`, ~23MB): Already loaded client-side via `@xenova/transformers` and cached by the browser. No change needed.
2. **SVC feature weights**: Exported as a JSON file, served as a static asset from `public/models/`. The service worker caches it with its existing cache-first strategy for assets. The file will be added to the service worker's `CACHE_URLS` for proactive caching on install.
3. **Classification at runtime**: Product name -> embedding (existing) -> apply SVC weights -> weighted cosine similarity against subcategories -> category. No server calls needed.

### Model Size Estimates

| Export Type | Size Estimate | Notes |
|------------|--------------|-------|
| Feature weights (384 floats) | ~2-5 KB | Just the learned dimension importances. Tiny. |
| Full weight matrix (105 x 384) | ~160 KB | If we want the full SVC decision function. |

The feature weights approach is preferred -- smallest possible payload, works with the existing subcategory matching, and is trivially extensible when more stores are added.

## Phases

### Phase 1: Embedding Generation

Generate embeddings for all 24,833 Wegmans products using the same model the browser uses.

**Scripts:**
- `generate_embeddings.py` - Uses `sentence-transformers` with `all-MiniLM-L6-v2` (same model as browser) to generate 384-dim embeddings for all products. Caches embeddings to `data/embeddings/` as `.npy` files. Also generates embeddings for the app's existing ~120 subcategory names (for baseline evaluation).

**Deliverables:**
- `data/embeddings/wegmans_product_embeddings.npy` (24,833 x 384 matrix)
- `data/embeddings/wegmans_product_labels.json` (category labels from scraped data)
- `data/embeddings/subcategory_embeddings.npy` (app's subcategory embeddings)
- Embedding generation report: time taken, dimensions verified

### Phase 2: Baseline Evaluation

Evaluate the current cosine-similarity approach to establish a baseline.

**Scripts:**
- `evaluate_baseline.py` - Reimplements the current `categoryClassification.js` logic in Python: same `CATEGORY_DEFINITIONS` subcategory list, same cosine similarity, same 0.25 threshold. Runs it against the Wegmans products and compares its predicted category against the scraped Wegmans category (using a lightweight automatic mapping for evaluation only). Reports accuracy.

**Why we still need a mapping for evaluation**: To measure accuracy, we need to compare what our classifier predicts vs what the product actually is. This is an evaluation-only mapping (Wegmans category -> expected app category) used solely to calculate metrics. It does NOT affect training.

**Deliverables:**
- Baseline accuracy report: overall accuracy, per-category precision/recall/F1
- Confusion matrix
- List of commonly misclassified products

### Phase 3: SVC Training

Train a Linear SVC on the Wegmans product embeddings using their native category labels.

**Scripts:**
- `train_svc.py` - Loads product embeddings and Wegmans category labels (105 classes). Splits 80/20 train/test. Trains `sklearn.svm.LinearSVC`. Extracts feature importance weights from the trained model's weight matrix (L2 norm across all class hyperplanes per dimension). Saves trained model and feature weights.

**How feature weights are derived:**
```python
# LinearSVC weight matrix: (105 classes x 384 features)
# Feature importance = L2 norm of each feature column across all classes
feature_weights = np.linalg.norm(svc.coef_, axis=0)  # -> (384,)
feature_weights = feature_weights / feature_weights.max()  # normalize to [0, 1]
```

**Deliverables:**
- Trained Linear SVC model (pickle, for reproducibility)
- Feature weights vector (384 floats)
- SVC accuracy on native Wegmans categories (train/test split)
- Feature weight analysis: which dimensions are most/least important

### Phase 4: SVC-Weighted Evaluation

Apply the learned feature weights to the cosine similarity approach and compare against baseline.

**Scripts:**
- `evaluate_svc.py` - Takes the feature weights from Phase 3 and applies them to the baseline approach: `weighted_embedding = embedding * feature_weights` before cosine similarity. Runs the same evaluation as Phase 2 with the weighted embeddings. Also tests product-to-product similarity improvements.

**Evaluation includes:**
- Category classification: SVC-weighted vs unweighted accuracy (same metrics as baseline)
- Product similarity: pick known similar products from Wegmans data (e.g., different parmesan cheeses), compare unweighted vs weighted similarity scores
- Analysis of which categories improved/degraded
- Go/no-go decision on browser integration

**Deliverables:**
- SVC-weighted accuracy report (same metrics as baseline)
- Side-by-side comparison: baseline vs SVC-weighted (categorization)
- Product similarity comparison: sample pairs showing improved similarity scores
- Go/no-go decision on browser integration

### Phase 5: Export and Browser Integration

Export the feature weights and update the classification service.

**Scripts:**
- `export_weights.py` - Exports the 384-float feature weight vector to `public/models/svc_weights.json`. Includes metadata (training date, source data, model version).

**Code changes:**
- `src/services/categoryClassification.js`:
  - On init: fetch and cache `svc_weights.json` (384 floats)
  - Before cosine similarity: multiply both product embedding and subcategory embeddings by the feature weights
  - Keep the existing subcategory matching flow (the categories and subcategories don't change)
  - Expose a `getWeightedEmbedding(productName)` method for future product similarity use
  - Fallback to unweighted similarity if weights file fails to load
- `public/sw.js` - Add `/models/svc_weights.json` to `CACHE_URLS` for proactive offline caching

**Future use (not in this plan):** The `getWeightedEmbedding()` method enables a future "duplicate detection" feature where the app compares a new product's weighted embedding against weighted embeddings of pantry items to suggest similar products. This would be built as a separate feature using the same SVC weights.

**Deliverables:**
- `public/models/svc_weights.json` (committed, <5KB expected)
- Updated `categoryClassification.js` with weighted similarity + `getWeightedEmbedding()` method
- Updated `sw.js` with weights URL in cache list
- Manual testing: classify sample products in browser, verify offline works

## Success Criteria

1. **Accuracy improvement**: SVC-weighted scoring has measurably higher accuracy than unweighted cosine similarity baseline when evaluated against Wegmans product data
2. **Model size**: Exported JSON is <10KB (just 384 floats + metadata)
3. **Offline works**: Weights load from service worker cache, classification works with no network
4. **Latency**: Per-product classification time is comparable to current approach (<50ms) -- the only added cost is one element-wise multiplication (384 floats)
5. **Graceful fallback**: If weights file fails to load, falls back to existing unweighted cosine similarity (no degradation)
6. **Extensible**: When more supermarket data is added later, re-running the SVC training produces updated weights with no code changes

## Notes

- **No category mapping needed for training**: The SVC trains on each store's native categories. The learned feature weights are domain-general -- they improve similarity scoring regardless of what target categories you compare against.
- **Mapping only for evaluation**: To measure whether the SVC actually improves classification, we need a lightweight mapping to compare predicted vs actual categories. This mapping is only used in the evaluation scripts, not in training or production.
- **Same embedding model**: Both current and SVC-weighted approaches use `all-MiniLM-L6-v2`. The SVC weights are a lightweight learned "lens" that focuses the embeddings on grocery-relevant features.
- **Two use cases, one set of weights**: The same 384-float feature weight vector improves both product categorization (product vs subcategory similarity) and product similarity (product vs product similarity). This enables the future duplicate detection feature for reducing food waste.
- **Linear SVC first**: A linear classifier on good embeddings is often surprisingly effective. We only explore more complex models if the feature weights don't improve accuracy.
- **No new browser dependencies**: Applying feature weights is a single element-wise multiplication (384 floats), implemented in plain JavaScript. No new ML libraries needed in the browser.
- **Additive with more data**: When HEB, Walmart, Kroger scrapers are done (v1 Phase 2), their data can be added to the training set. More diverse product-category examples will produce better-calibrated feature weights. The pipeline (generate embeddings -> train SVC -> export weights) stays the same.
- **Food waste reduction**: The accuracy of duplicate detection scales directly with the quality of these learned weights. More training data from diverse stores = better weights = better duplicate detection = less food waste.
