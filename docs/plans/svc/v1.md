# Support Vector Classifier for Product Categorization - Plan v1

## Goal

Train a Support Vector Classifier (SVC) with proper feature weights to **replace** the current cosine-similarity-only classification in SuperSuper. To build the training dataset, we will scrape product names and their full category hierarchies from multiple supermarket websites.

### Why This Matters

The current `categoryClassification.js` uses a manually curated set of ~120 subcategories mapped to 12 top-level categories with zero-shot cosine similarity against `all-MiniLM-L6-v2`. This approach lacks real-world weighting - every subcategory has equal influence and the model has no understanding of how products actually relate in grocery stores.

For example, Wegmans categorizes "Parmigiano Reggiano Cheese" under `Departments > Cheese > Hard Cheeses` - a 3-level hierarchy that captures product relationships the current flat list misses.

By scraping real category trees from multiple supermarkets, we can:
1. Build a training dataset with real-world product-to-category mappings
2. Train an SVC that learns proper feature weights from real grocery data
3. **Replace** the current cosine-similarity classification with the trained SVC model

Additional supermarkets can be added later by creating a new standalone script following the same pattern.

### Walmart: In-Store Products Only

Walmart's online catalog includes products from third-party/marketplace sellers that are only available for shipping and are **not physically present in store aisles**. Since this data is for an in-store shopping experience, the Walmart scraper must:

- **Only include products sold and fulfilled by Walmart** (not marketplace/third-party sellers)
- Filter using available seller/fulfillment indicators in the API or product data (e.g., "Sold & shipped by Walmart", "In-store" availability flags)
- Exclude products marked as "Online only", "Shipped by [third-party]", or similar
- If the API provides a `seller` or `fulfillmentType` field, filter to first-party Walmart items only

This filtering requirement may also apply to other supermarkets if they have marketplace models in the future. Each scraper should document any seller/availability filtering it applies.

## Data Schema

The unified schema must accommodate different hierarchy depths across supermarkets.

### Product Record

```json
{
  "source": "wegmans",
  "product_name": "Wegmans Italian Classics Parmigiano Reggiano Cheese",
  "category_hierarchy": ["Departments", "Cheese", "Hard Cheeses"],
  "hierarchy_depth": 3,
  "source_url": "https://www.wegmans.com/shop/categories/4459767",
  "scraped_at": "2026-03-04T01:30:00Z"
}
```

### Key Design Decisions

- **`category_hierarchy`**: An ordered array from broadest to most specific. Different supermarkets will have different depths (e.g., Wegmans may have 3 levels, Walmart may have 4).
- **`product_name`**: The full product name as displayed on the site, which may include brand, size, and variety. We keep it raw and handle parsing downstream.
- **`source`**: Identifies which supermarket the data came from, enabling per-source and cross-source analysis.
- **No price/weight fields**: We are focused on category relationships, not pricing. Brand information is embedded in the product name and can be extracted later if needed.
- **In-store filtering**: For supermarkets with marketplace sellers (e.g., Walmart), only products physically available in store aisles are scraped. External/third-party seller products are excluded at scrape time.

### Output Format

- Primary output: **JSON Lines** (`.jsonl`) - one product per line, easy to stream and append
- One file per supermarket per scrape run: `data/{source}_{date}.jsonl`
- A merged/deduplicated dataset for SVC training: `data/merged_products.jsonl`

### Scraping Approach

Each supermarket will likely need one of these approaches, chosen per-script:

| Approach | When to Use | Tools |
|----------|-------------|-------|
| **API-based** | Site has a public/semi-public API | `requests` |
| **HTML parsing** | Server-rendered pages | `requests` + `BeautifulSoup` |
| **Browser automation** | JS-rendered / SPA sites | `Playwright` |

Most modern grocery sites are SPAs that load data via internal APIs. The preferred approach is to identify these API endpoints (via browser DevTools) and call them directly - this is faster and more reliable than parsing HTML.

### Rate Limiting and Politeness

Each script should include:
- Configurable delay between requests (default: 2-3 seconds)
- Respect `robots.txt`
- User-Agent header identifying the scraper
- Basic retry logic with backoff on errors
- Stop on HTTP 429

## Notes and Considerations

- **Legal/ethical**: Scraping should respect each site's Terms of Service and `robots.txt`. This data is for personal/research use in a self-hosted application.
- **Data freshness**: Supermarket catalogs change frequently. Scripts are designed to run once but can be re-run to refresh data.
- **Schema evolution**: The JSONL format allows adding fields later without breaking existing data. If we need additional fields (price, brand parsed out, product image URLs), we can add them in v2.
- **SVC replaces cosine similarity**: The trained SVC model from Phase 4 will **replace** the current zero-shot cosine similarity approach in `categoryClassification.js`. The `CATEGORY_DEFINITIONS` and embedding comparison will be swapped out for SVC-based classification with proper learned weights.
- **Training data**: The `scripts/scraper/data/` directory should be gitignored (scraped data can be large). Only sample outputs and the schema documentation should be committed.
- **Simplicity**: Each data collection script is standalone. No class hierarchies, no shared base modules. Copy-paste common patterns between scripts - this keeps each one easy to understand, modify, and debug independently.
